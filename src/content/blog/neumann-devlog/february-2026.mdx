## February 2026 — The Rewrite
</div>

The decision to ditch GPUI took about a day to make and about a week to feel good about. The replacement architecture: a Rust server running on loopback, a Next.js browser UI served from the same origin, talking to each other over a WebSocket.

The thing that unlocked this decision was accepting a split I'd been resisting: the backend and the frontend don't have to be the same language. The Rust side owns what Rust is good at — state, processes, filesystem, concurrency. The browser side owns what browsers are good at — rendering, hot reload, layout, DevTools. The two halves talk through a typed protocol. That's not a compromise, it's just a good architecture.

I bound the server to loopback only (`127.0.0.1:8421`) and made that a hard constraint. No authentication, no remote access, no multi-user anything. That one decision quietly eliminated an entire class of problems — security hardening, session management, concurrent writers, distributed state. Localhost-only means I could focus on building the tool instead of building infrastructure around the tool.

<Note avatar="https://i.imgur.com/ZF1UFR0.jpeg">
  Constraints that feel like giving up are often the most liberating design decisions. "Loopback only" wasn't a compromise — it was choosing a scope that let me actually ship things.
</Note>

## The Cloud Layer

The first thing I needed was a way to talk to LLMs. I built `neumann_cloud` — a routing layer that sits between the UI and the model providers (OpenAI, Anthropic, Gemini). One place that handles provider-specific quirks: different API formats, different streaming protocols, different tool call schemas.

Early on I had each provider's tool definitions duplicated separately. That was a mistake that bit me almost immediately — the three lists diverged within days, and fixing a tool schema meant fixing it in three places. Centralizing them into a single canonical list was one of the first real architectural improvements.

The Anthropic integration specifically required handling stream keepalive pings. Anthropic sends periodic ping events during long-running tool calls to keep the connection alive. Without handling them, any agent turn longer than ~30 seconds would silently time out and lose the response. That was a frustrating bug to debug — the agent would "finish" with no output and no error.

## The Model Discovery Problem

The original UI had a hardcoded list of model names for OpenAI and Gemini. This worked until I added Anthropic, at which point I had three separate hardcoded lists that needed to stay in sync with each other and with whatever the providers actually had available.

The fix was to make the server discover available models dynamically from each provider's API and surface them to the UI. The UI renders whatever the server reports — no hardcoded names, no fallback lists. When a provider adds a new model, it appears automatically. This is the only way this could work long-term.

<IssueList>
  <Issue title="Token double-counting">
    The chat panel accumulated per-turn token usage at two separate points in the render cycle, inflating the displayed totals. Fixed by consolidating tracking into one place using <code>tracing</code> spans.
  </Issue>
  <Issue title="Stream keepalive missing">
    Anthropic sends keepalive pings during long tool-use turns. Without handling them the HTTP connection would close after ~30 seconds and the agent response was lost silently — no error, no output, just nothing.
  </Issue>
  <Issue title="Model list hardcoded">
    Three separate lists of model names, one per provider, all drifting out of sync. Switched to server-driven dynamic discovery. The hardcoded lists were removed entirely.
  </Issue>
  <Issue title="Tool definitions diverged across providers">
    Each provider had its own copy of the tool schemas that had evolved independently. Centralized into a single list that all providers read from — one place to update when a tool changes.
  </Issue>
</IssueList>

By the end of February the thing worked. Not polished, not fast, but it would take a task, run the agent, stream the response, and show you the output. That was the foundation. March was when it actually started to look like a product.

<Note avatar="https://i.imgur.com/ZF1UFR0.jpeg">
  "It works" is a humble milestone but an important one. January ended with something that compiled and accumulated bugs. February ended with something that did the thing it was supposed to do. That's a real difference.
</Note>
