  ## January 2026 — GPUI, VCS, and a Dead End
</div>

January started with a refactor and ended with a decision to throw most of it away.

The Jan 4 commit split the project into separate Rust crates — `neumann_domain`, `neumann_server`, `neumann_backend` and so on. That was the right architectural move. A monolithic crate is fine when a project is small; as soon as you want separate compilation units, clear dependency boundaries, and the ability to test the domain layer in isolation, you need the split. I did that early and it paid dividends for the rest of the project.

Then I used an AI agent (ralph) to implement VCS features — git staging, status badges in the file tree, a diff viewer, status refresh with debouncing. This was an experiment in using an agentic coding loop for a non-trivial feature. It worked, mostly. The commits landed and the features were functional. But the GPUI renderer underneath was still the bottleneck.

## The GPUI Problem

The deeper issue was the framework itself. GPUI is built _for Zed_. It's not a general-purpose UI toolkit — the abstractions are shaped around how Zed works internally, and the documentation assumes you already understand Zed's architecture. Every fix I made felt like I was reading the source of a codebase I didn't own and guessing at which lever to pull.

The feedback loop was punishing. GPUI compiles with the rest of the Rust binary — every UI tweak meant a full `cargo build`, 20 to 40 seconds per cycle. For layout iteration, where you want to see the result of a padding change in milliseconds, that's brutal. Browsers solve this with hot reload. GPUI does not.

By the end of January the issues had accumulated faster than I was fixing them: streaming markdown not rendering in real time, WebSocket state going stale after a refactor, cursor invisible on light backgrounds, layout not reflowing correctly. None catastrophic individually. Together they pointed at something structural.

## The Decision

The decision to scrap GPUI came down to one question: how much of my time was I spending building the editor versus fighting the framework?

The answer was too much. The browser is already a great UI runtime — hot reload, flexbox that works, a debugger that isn't fighting you, DevTools, and a decade of tooling refinement. Fighting all of that to stay in a single language wasn't principled engineering. It was stubbornness.

<Note avatar="https://i.imgur.com/ZF1UFR0.jpeg">
  The VCS work survived the rewrite almost entirely. The January commits weren't
  wasted — the domain types and git logic ported over cleanly because I'd put
  them in their own crate, isolated from the UI. Good architecture paying off
  earlier than expected.
</Note>

February was a full rewrite. Rust server on loopback, Next.js browser UI, WebSocket between them. The VCS work from January survived — the domain types and git logic ported over cleanly, because the crate split in January had made them independent of the UI layer.
