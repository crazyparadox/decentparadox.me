  ## March 2026 — The IDE Takes Shape
</div>

February gave me a working agent loop. March turned it into something I'd actually want to use every day.

The mental shift that drove everything in March: stop thinking about Neumann as a "chat UI with some tools" and start thinking about it as a proper local IDE. That reframe changed the whole roadmap. A chat UI needs a text box and a message thread. An IDE needs a file tree, a git history panel, an inline editor, a terminal, a project switcher, a CLI, a way to install it as a desktop app.

These weren't features I added one by one out of boredom. Each one came from hitting a wall in my actual workflow. I'd use Neumann to run an agent task, it would edit some files, and then I'd have to open my regular editor to see what changed. So I needed an inline file editor. The agent would run git commands and I'd have no way to see the history. So I needed a git panel. I wanted to use it from the terminal without a browser window. So I needed a CLI.

The whole month was: use the thing, find what's missing, build it.

## Becoming an IDE

The biggest single day was March 11. I shipped the IDE layout redesign, the file tree, the git history panel, the inline file editor, full-text search with syntax highlighting, file type icons, keyboard shortcuts, and a new onboarding flow — all in one push. Most of these had been in progress for a week; March 11 was when they all got merged and stabilized together.

The file tree is server-driven. The browser doesn't have direct filesystem access — it asks the Rust process for directory listings. This is the right design for a localhost app: the server owns the filesystem, the browser renders it. It also meant I could add file watching and live-updating the tree later without touching the frontend at all.

The inline editor was the feature I was most nervous about. Building a full code editor from scratch is a multi-year project. I didn't build one — I built a viewer/editor that handles the common cases: open a file, read it, make changes, write it back. For anything more complex the agent handles the editing. That turned out to be the right scope.

## The CLI and TUI

Around March 21 I shipped the `neumann` CLI — a binary that opens a TUI session backed by the same cloud engine as the browser UI. I wanted to use Neumann from inside a terminal without opening a browser window. Sometimes you're SSH'd into a machine. Sometimes you want to script something. A CLI gives you that.

The TUI ended up with some features I really liked. `@file` syntax injects file contents as a structured attachment — type `@src/main.rs` in the composer and it gets sent to the agent alongside your message. `!cmd` runs a shell command and shows the output. `!!cmd` runs a command and injects the result into your message, so you can do things like `!!git diff` and have the diff automatically included in your prompt.

By March 27, `tui.rs` had grown to 3,060 lines. That's the point where a file stops being a file and starts being a liability. Every change requires scrolling past hundreds of lines of unrelated code, merge conflicts become painful, and the structure gets lost. I split it into focused modules: `layout.rs`, `render.rs`, `events.rs`, `messages.rs`, `editor.rs`. The split took a few hours and immediately made everything easier to work in.

## SQLite

Before March, conversation history was stored in memory. Every server restart, your threads were gone.

Switching to SQLite was one of those changes that felt obviously right in retrospect but took a deliberate decision to do. The implementation was straightforward — `rusqlite` with bundled SQLite, WAL mode, a migrations table, UUID thread IDs, atomic load+save transactions. One afternoon of work. Conversations survived restarts. I immediately wondered why I'd waited this long.

<Note avatar="https://i.imgur.com/ZF1UFR0.jpeg">
  SQLite is almost always the right answer for this problem space. One file, no
  server, survives restarts, readable with any tool. I should have reached for
  it on day one instead of treating in-memory state as a temporary solution that
  somehow lasted two months.
</Note>

The SQLite work also led to hardening the server's path handling. Issue #15 was a path traversal bug: `resolve_workspace_path` was following symlinks without checking if the resolved path escaped the workspace root. A crafted symlink could reach anywhere on the filesystem. Fixed with explicit symlink resolution and boundary checks — resolve the symlink, verify the result is still inside the workspace root, reject if not.

## Tauri

I added Neumann to a Tauri desktop wrapper around March 23. The server now starts in-process when the desktop app launches and the UI loads in a WebView pointed at the local server. You can install Neumann as a `.app` and use it like a native app — no terminal required, no port to remember.

The Tauri integration also introduced its own bugs. Stale Claude session IDs were one: after a Tauri restart, the cached session ID was expired server-side. All requests would fail silently until the session was cleared — no error message, no indication anything was wrong, just nothing happening when you expected something. Fixed by detecting the stale-session error response and triggering re-authentication.

By the end of March, Neumann was something I'd genuinely use as my primary tool. A file tree, a terminal, a git panel, inline editing, a CLI, a desktop app. It wasn't finished, but it was real. April was when I started shipping features at a pace I couldn't quite keep up documenting.
